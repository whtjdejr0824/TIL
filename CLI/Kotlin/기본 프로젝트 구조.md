# 기본 프로젝트 구조

프로젝트 : 코틀린으로 어플리케이션을 짤 때, 관련한 모든 내용을 담는 '큰 틀'

모듈 : 하나의 프로젝트는 여러개의 모듈로 이뤄질 수 있음.

'직접' 만들 수도 있고, 필요한 기능을 이미 구현해 둔 '라이브러리 모듈'을 가져와 붙일 수도 있기 때문에 굉장히 편리한 기능 단위

모듈안에는 다수의 폴더와 파일이 들어 갈 수 있음. 여기에는 코틀린 코드파일(.kt)뿐만 아니라 모듈과 관련된 설정 및 리소스 파일 등도 포할 될 수 있음.



프로젝트, 모듈, 폴더 및 파일이 실제 파일 시스템에 기반한 물리적인 구조를 담당한다고 하면

논리적인 구조로는 패키지라는 것이 있음.

개발시에 소스 코드의 '소속'을 지정하기 위한 논리적 단위

코드를 작성할 때는 코드내에서 사용하는 이름이 용도에 따라 서로 충돌하지 않도록 유니크한 패키지이름을 지정하는 것이 중요.



일반적으로 패키지의 이름을 지을 때는 서비스 도메인을 거꾸로 배열하고, 그 뒤에 프로젝트명을 붙임.

그 밑에 기능별 세분화 하여 붙임



코드 파일을 패키에 넣는 방법

```kotlin
package com.youtube.dimo // 패키지를 명시하지 않으면 자동으로 'default'패키지로 묶이게 되는 것도 알아둬야함.
import com.youtube.dimo.base // 사용할 외부 패키지 이름을 써주면, 변수, 함수, 클래스 사용 가능

fun main() {
    
    println("프로젝트의 구조는 재밌어")
    
}
```

코틀린은 자바와 달리, 폴더 구조와 패키지 명을 일치시키지 않아도 됨.

단순히 파일 상단에 패키지명만 명시해주면 컴파일러가 알아서 처리하게 됨

그런데 같은 패키지 내에서는 변수 함수 클래스를 공유할 수 있는데, 서로 이름만 부르면 사용가능.

패키지가 다르면 import라는 작업을 해야 사용가능



코틀린은 클래스명과 파일명이 일치하지 않아도 되며, 하나의 파일에 '여러개의 클래스'를 넣어도 알아서 컴파일이 가능.

이는 파일이나 폴더를 기준으로 구분하지 않고, 파일내에 있는 package키워드를 기준으로 구분하기 때문



이 구조는 코틀린으로 실제 어플리케이션을 만들 때 뿐만 아니라, 접근제한자에서도 사용되는 개념.
